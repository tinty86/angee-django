# Vetted spec — design-DRY cleanup

> Reference for implementation. Apply the plan's corrected DECISIONS where they override this (blocker fixes). Verdict: needs-changes.

## Summary
This stream closes the genuinely-open color/radius/typography DRY gaps left after the merged react-consistency two-axis-color work, without redoing it. It establishes/uses these owners: (1) `lib/tones.ts` becomes the single source for tone-colored TEXT via a new `toneText(tone)` accessor (sibling to the existing `toneSolidBg`/`toneGlyph`) plus a `FEEDBACK_INTENTS` constant, so the five hand-typed per-tone text maps source their literals from one place; (2) the foreground `*-text` tokens are removed from solid-BACKGROUND positions and routed through `toneSolidBg`/`bg-<tone>`; (3) the phantom `text-brand-text` class is replaced with the real `text-brand-soft-text`; (4) the avatar gradients are rebuilt from existing Angee primitive ramps instead of undefined `--color-*` refs + raw-hex fallbacks; (5) one canonical radius spelling (the Angee token scale `rounded-N`) replaces the three 6px spellings + the stray Tailwind named scale; (6) a new `iconTileVariants`/`IconTile` recipe owns the icon-chip class duplicated across the metric/header fragments; (7) a small `textRole` recipe set names the title/caption/meta roles. The owner model: tones.ts owns every tone→class literal (bg via toneSolidBg, text via toneText, glyph via toneGlyph, full pill via toneClass); the @theme/token layer owns radius/size literals; recipes (button, popover, IconTile, textRole) own component class strings; call sites compose recipes, never re-spell literals.

## Risks
- Radius codemod scope: a blanket `\brounded(?=[ "'])` regex would corrupt the `shape:{rounded:"rounded"}` variant keys/refs in badge.tsx/chip.tsx and the `rounded:` classGroup key in tailwind-merge-config.ts. Mitigation: EXCLUDE badge.tsx, chip.tsx, tailwind-merge-config.ts, button.tsx, popover.tsx, dialog.tsx from the bare-rounded pass and edit those by hand (the `rounded-md/sm/lg` suffix passes are safe everywhere since the suffix avoids key collisions).
- tailwind-merge corner subgroup gap: `rounded-t-10` (new) is not registered in tailwind-merge's `rounded-t` subgroup, so `cn("rounded-t-10","rounded-t-2")` may not de-dupe. Today only BoardView uses a single rounded-t per element, so no live conflict; flag as a follow-up (extend the config with rounded-t/b/l/r token subgroups if corner conflicts ever appear).
- text-2xs vs leading: AppChooser:268 `text-[11px] leading-[1.125rem]` (11px/18px). text-2xs bundles 16px line-height; the explicit `leading-[1.125rem]` must be KEPT so the visible line-height does not regress to 16px.
- toneText correctness across tones: the brand/accent/purple/pink rows use `-soft-text` not `-text`, so `toneText` must mirror the soft-row text per tone (not a naive `text-<tone>-text`). The 5 consumers only use the feedback subset where it equals `text-<tone>-text`, but the owner must still be palette-correct for future callers.
- tv key inference: building variant maps via `Object.fromEntries(...)` loses literal keys and would widen CodeTone/StatusIconTone to `string`. Mitigation: keep explicit keys and call `toneText("info")` per entry (as specified) rather than spreading.
- IconTile glyph sizing is coupled to tile size (sm/md=3.5, lg=4); encoding it in the size variant is correct, but verify no current consumer relied on an off-pairing.
- Visual parity: avatar gradient hues shift slightly (e.g. orange→red, indigo→brand) since the undefined ramps are replaced by Angee primitives; this is intentional (token discipline) but is a visible change — spot-check both themes.
