## Summary
Today `widgetControlSurfaceVariants` (packages/ui/src/ui/widget-control.ts) shares only the FLATTENED surface string: input.tsx:13/28 aliases it then flattens `INPUT_SHELL = inputSurfaceVariants({ focus: "self" })` to a class string, and textarea/select/number-field don't touch the recipe at all — they import only the `WIDGET_CONTROL_READONLY_CLASS` constant. The recipe's own `invalid`/`readOnly`/`disabled` VARIANTS are effectively dead, while every control RE-DECLARES `invalid` (9 sites: input.tsx:45 & :102, textarea.tsx:21, select.tsx:58, number-field.tsx:57, field.tsx:21 & :67, checkbox.tsx:22, widget-control.ts:28) with drifting focus-pseudo coverage (focus: vs focus-visible: vs focus-within:) and `readOnly` ~6x. Separately, `outline-none transition-colors focus-visible:focus-ring disabled:cursor-not-allowed disabled:opacity-{50|60}` is copy-pasted across button.tsx:12, toggle.tsx:10, collapsible.tsx:17, toolbar.tsx:18/20 and the form controls, with the opacity forking 50 vs 60 (and 40 on number-field steppers). `tailwind-variants` (3.2.2 installed, ^3.2.0 in package.json) `extend`/`compoundVariants`/`compoundSlots` are used in ZERO recipes today. The owner model this stream establishes: (L0) a new `interactiveSurfaceVariants` owning focus-ring + disabled dimming, extended by button/toggle/collapsible/toolbar + all form controls; (L1) `widgetControlSurfaceVariants` `extend`s L0 and owns the box chrome (border, focus-border, invalid danger via focus-aware compoundVariants, readOnly-transparent, disabled bg-inset, a `surface` variant); (L2) non-slot controls (input, textarea) `extend` L1 so invalid/readOnly/disabled FLOW from the owner; slotted controls (number-field.group, field.controlFrame, select.trigger, searchInput.root) COMPOSE `widgetControlSurface({...})` onto their one chrome slot — because `extend` of a non-slot parent does NOT route onto a named slot (verified). The pattern already exists and proves the design: widgets/markdown.tsx:103 already calls `widgetControlSurface({ focus:"within", readOnly })`. Rendered class output stays equivalent (tailwind-merge de-dupes); the only flagged visual deltas are the disabled-opacity convergence and select's `rounded`->`rounded-md`, both overridable or sign-off items.

## Risks
- extend cannot route a non-slot parent onto a named slot: VERIFIED via scratchpad probe (a slotted child extending a non-slot parent puts the parent contribution on an implicit `base` slot, NOT on `trigger`/`group`/`root`). This is why slotted controls (number-field, field, select, searchInput) must COMPOSE `widgetControlSurface({...})` at the call site rather than `extend`. Mis-applying extend there would silently drop the chrome.
- Cascade ordering for invalid vs surface border: the resting `border-danger` MUST come after the surface `border-border` in the merged string or tailwind-merge keeps the wrong one. Solved by putting the danger in compoundVariants (applied last) — VERIFIED. If a future edit moves the danger back into a plain `invalid` variant declared before `surface`, the resting red border regresses to neutral.
- invalid+readOnly interaction: readOnly must suppress the danger ring. Achieved by guarding every danger compoundVariant on `readOnly:false` AND by WIDGET_CONTROL_READONLY_CLASS forcing border/focus transparent — VERIFIED. Drop either guard and a read-only invalid field shows a red border.
- Mechanism mismatch on `disabled`: native inputs disable via the `:disabled` pseudo; base-ui Group/Trigger/Toolbar.Button via `data-[disabled]`. The owner exposes `disabled: pseudo|data|none` for this; picking the wrong mode silently disables the dimming. Verify each control's actual disabled attribute before choosing the mode.
- Public-API trim: INPUT_SHELL/INPUT_SHELL_WITHIN/INPUT_BASE/TRIGGER_BASE/INPUT_READONLY/INPUT_NUMERIC/inputTriggerVariants/inputSurfaceVariants are re-exported via `export * from ./ui/input` in index.ts. Monorepo grep found zero consumers, but external (downstream addon) consumers cannot be ruled out from this repo. Deleting them is the prefer-deletion call; if downstream breakage is a concern, keep thin aliases for one release.
- searchInput multi-surface + select inset divergence: these controls do not use the plain sheet surface, so they map onto the new `surface` variant (inset/plain) and keep a couple of local overrides (select `rounded`, searchInput `focus-within:bg-sheet`). Each override is an equivalence point that must be diffed, not assumed.
- field.control uses data-[invalid] (attribute-driven) not a prop — its danger stays local. If a later refactor tries to route it through the prop-driven `invalid` variant the invalid styling will never fire (no prop is passed).
