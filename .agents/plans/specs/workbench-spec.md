# Vetted spec — console-shell Workbench

> Reference only. The PLAN's decision **D1 overrides this**: create `Workbench.tsx` AND
> migrate every Explorer consumer (storage/knowledge) to it, then DELETE Explorer (one owner).
> Use this spec for file-level detail on Chatter fold, SplitPanes v4 collapse handle, barVariants,
> grid simplification, and the addon migrations. Verdict: needs-changes.

## Summary
Establish ONE inner shell owner — a new `layouts/Workbench.tsx` — that composes the existing `page/SplitPanes` (v4) into the VS Code/Zed slot vocabulary: primary sidebar | (content / optional bottom) | secondary sidebar, every pane collapsible + resizable + size-persistent. ConsoleLayout stops being a fixed CSS grid for the side regions: it keeps owning the fixed chrome rows (rail, topbar, crumbs, control, status) but hands the content+sidebar+chatter region to the Workbench, so the `area-sidebar`/`area-chatter` grid columns and the `console-grid-sidebar` variant are deleted. Owner model: (a) the LIBRARY (react-resizable-panels via SplitPanes) owns sizing/collapse/persistence — no hand-rolled pointer/arrow resize survives; (b) the Workbench owner owns the nested-Group composition and slot vocabulary; (c) `chatter-context` is demoted from holding ephemeral pixel width to being only the cross-tree bridge that lets the chrome TopBar toggle drive the secondary pane's collapse controller; (d) ONE `barVariants` (tv) recipe owns bar chrome (height/padding/border/bg/tone) that the 8 bars compose, with `createLayoutBand` kept only for the two portal bands. SchemaPage and AgentSessionsPage stop hand-rolling fixed 3-pane/`w-60` shells and adopt the extended Explorer. SplitView/RecordView/CanvasPage (zero production consumers) are deleted. Naming is normalized: slot Symbol ids move from `@angee/base.*` to `@angee/ui.*`, and Pane (split region) vs Aside (page side content) vs primary/secondary (workbench sidebars) get one rule each. Net effect: the Chatter persistence bug (resize+reload resets) is fixed for free because the secondary pane's size+collapse persist through `useDefaultLayout`/localStorage exactly like Explorer's panes already do.

## New files
- `/Users/alexis/Work/angee/angee-django-ds-workbench/packages/ui/src/layouts/Workbench.tsx` — The new inner-shell owner. A function component `Workbench` plus `Workbench.Primary`/`Workbench.Content`/`Workbench.Secondary`/`Workbench.Bottom` slot sub-components (Object.assign pattern, mirroring SplitView's slot shape but with @angee/ui symbol ids via createLayoutSlot). Composes nested SplitPanes Groups: an OUTER horizontal Group [primary | center | secondary] where `center` is an INNER vertical Group [content / bottom]. Each side/bottom pane uses SplitPane with `collapsible`, a `defaultSize`, `minSize`, and the shared border/bg classes proven in Explorer (`border-r/border-l border-border-subtle bg-sheet-2`). Props: `primary?`, `secondary?`, `bottom?`, `children` (=content), `autoSave?` (persistence id; default 'workbench'), `primarySize?`/`secondarySize?`/`bottomSize?` (percent defaults), `primaryCollapsed?`/`secondaryCollapsed?` (initial), `className?`. Panes render only when supplied (Explorer's omit-pattern: with no primary/secondary/bottom it is a bare content frame, no resize machinery). Each collapsible pane is wired to a `useCollapsiblePane` controller and publishes it upward so a sibling chrome toggle can drive it: the secondary controller registers into chatter-context; the primary controller is exposed for a wB-style sidebar toggle. The single composition the whole console flows through.

## Risks
- v4 isCollapsed() is non-reactive: a toggle button's pressed state must be derived via onResize (and re-derived on drag-to-collapse), not by reading the imperative handle in render. useCollapsiblePane must own this or the TopBar chatter glyph desyncs.
- Cross-tree bridge fragility: TopBar (chrome) and the secondary pane (layouts/Workbench) are siblings under ConsoleLayout, bridged through chatter-context. If controller registration order or unmount cleanup is wrong the toggle silently no-ops. Need a fallback to local collapsed state when no Workbench is mounted (standalone/tests/Storybook TopBar).
- Persistence-key migration: existing users have chatter width in React state only (never persisted) and Explorer panes under their own autoSave ids. The new console.workbench/iam.schema/agents.sessions ids are greenfield, but ensure SplitPanes' per-id remount (key=autoSave, lines 113-115) doesn't fight the nested Groups (outer + inner each need a stable distinct id, or a single Group layout).
- Layout regression from grid simplification: moving Chatter out of the area-chatter column (which spanned crumbs+control+content rows) into a content-region pane means the secondary sidebar no longer sits beside the breadcrumb/control band. This is the intended VS Code model but is a visible change; confirm crumbs/control remaining full-width above the workbench reads correctly.
- Bottom Panel is net-new with zero adopters — adding it risks an unused abstraction (violates 'prefer deletion to abstraction'). Spec it optional/deferred unless an adopter (e.g. agents transcript/output) is committed.
- Deleting RecordView/CanvasPage/SplitView removes their slot Symbols and stories; verify no non-storybook import remains (grep confirmed only stories import them) and update console-design-system.md's table references so docs don't drift from code.
- SessionRail and SchemaPage panels currently supply their own bg/border/rounded chrome; moved into panes the pane supplies bg/border. Double-borders or lost rounding appear if the inner chrome isn't stripped.
- Nested SplitPanes Groups: react-resizable-panels requires at least one panel with preserve-relative-size in each Group; the inner vertical content/bottom Group must satisfy this or it throws.
