# Operator daemon → a Refine `operator` data provider (retire the urql quarantine)

> Status: **proposed — research-backed design, not yet scheduled.** This file is the
> plan; execution is gated on the architect picking the provider-contribution seam
> (§7, Open decision A). Related: `refine-adoption-refactor-plan.md`,
> `refine-revolution-pass-ledger.md`, `hasura-row-resource-non-model.md`.

## 1. The question and the reframe

The starting question was: *"can the operator daemon be changed to align better with
Hasura, so it stops being an edge case?"*

Research answer: **the daemon already speaks Angee's Hasura wire contract.** It is not a
bespoke RPC schema that needs reshaping. The edge case is **not the daemon's schema** —
it is that the **frontend consumes the daemon through a parallel urql stack** instead of
as a Refine data provider like every other schema.

So this is not "make the daemon more Hasura." It is "**stop ignoring the alignment that
already exists** on the consumption side."

### Evidence

- **Daemon SDL is hasura-default for its resources.** `operator/web/schema/operator.graphql`
  (generated by `manage.py operator_schema`, introspected into the console codegen)
  exposes the full Hasura triplet for `services`/`jobs`/`sources`/`workspaces`/`templates`/
  `secrets`: `xxx(where: xxx_bool_exp, order_by, limit, offset)`, `xxx_by_pk(id: String!)`,
  `xxx_aggregate`, `insert_xxx_one`, `update_xxx_by_pk(pk_columns, _set)`,
  `delete_xxx_by_pk`, with `_eq/_ilike/_in/_is_null` operators and `idType: String`. This is
  the same contract `docs/stack.md:85` says the daemon shares with `strawberry-django-hasura`
  and `@refinedev/hasura`.
- **Several frontend mutations are already plain Hasura CRUD:** `OperatorServiceCreate →
  insert_services_one`, `OperatorServiceDestroy → delete_services_by_pk`,
  `OperatorWorkspaceCreate/Destroy`, `OperatorSecretSet/Delete`
  (`operator/web/src/data/documents.daemon.ts`).
- **The code already names this as the intended direction.** `operator/web/src/data/
  operator-client.ts:20` — *"The operator owns this daemon urql quarantine **until it is
  rebuilt on the refine provider.**"* This plan is that rebuild.
- **The provider primitive already accepts what we need.** `createAngeeHasuraDataProvider`
  (`packages/refine/src/provider.ts:84`) takes `auth?: AuthFetch`; `bearerAuth(token)`
  already exists (`packages/refine/src/transport-auth.ts:74`). The only missing piece is a
  token-getter variant (§5).

## 2. Consumption shape (what we are actually migrating)

Frontend inventory of `operator/web/src/`:

| Surface | Count | Nature |
|---|---|---|
| Live GraphQL subscriptions | 3 | `onStackSnapshotChange` (whole-stack push), `onWorkspaceLogs`, `onWorkspaceStatusChange` |
| One-shot queries | 3 | `SNAPSHOT_QUERY` (first paint), `SERVICE_ENDPOINT_QUERY`, `WORKSPACE_LOGS_QUERY` (history) |
| Mutations | 25 | service/stack/source/workspace/secret/job lifecycle — mix of Hasura CRUD and operational RPC |
| Raw WebSocket | 1 | `/services/{name}/logs/stream` (non-GraphQL) |

The dashboard panes do **not** fetch per-resource. They render from one zero-variable
`STACK_SNAPSHOT_SUBSCRIPTION` that pushes the entire stack state, deduped across all 8 panes
(`operator/web/src/data/transport.tsx:208` `useOperatorSnapshot`). This is a deliberate,
superior design for live monitoring — one socket, one payload, rendered directly.

## 3. Scope decision — what moves, what stays special

**Moves onto a Refine `operator` data provider (request/response only):**

| Today | Target | Fit |
|---|---|---|
| `useOperatorAction` (`insert_*_one`/`delete_*_by_pk` docs) | `useCreate` / `useDelete`, `dataProviderName:"operator"` | exact |
| `useOperatorAction` (command RPCs: `stackUp`, `serviceStart`, `sourcePull`, `jobRun`, git ops) | `useCustomMutation` authored op (= app's `useActionMutation`/`useAuthoredMutation` pattern) | good |
| `SERVICE_ENDPOINT_QUERY` via direct urql `useQuery` (`ServiceDetail.tsx:21`) | `useAuthoredQuery`/`useCustom`, `dataProviderName:"operator"` | exact |
| `WORKSPACE_LOGS_QUERY` history via direct urql `useQuery` (`logs.tsx:60`) | `useAuthoredQuery`/`useCustom`, `dataProviderName:"operator"` | exact |

Those two `useQuery` imports are the original "edge case" finding — they vanish here.

**Stays special — and that is correct, not a compromise:**

1. **The whole-stack snapshot push** (`onStackSnapshotChange`) — a render-stream, not
   resource-change events. Keep it on the daemon's own graphql-ws transport. Refine's live
   provider is an *invalidation* channel, and the root `<Refine>` already has exactly one
   liveProvider (the console change stream) — Refine allows only one. So daemon subscriptions
   do **not** become a second root liveProvider.
2. **GraphQL log/status subscriptions** (`onWorkspaceLogs`, `onWorkspaceStatusChange`) — same:
   render-streams on the daemon ws transport.
3. **The raw log WebSocket** (`/services/{name}/logs/stream`, `logs.tsx:162`) — not GraphQL;
   has no Refine home. Unchanged.

**Explicitly do NOT** convert the dashboard panes to per-resource `useList` + resource
liveProvider. That would shatter one snapshot push into N subscriptions + N invalidate→refetch
round-trips — a regression. The panes keep the snapshot-push model.

Net effect: the "edge case" shrinks from *"an entire parallel urql data stack (client + cache
+ fetch exchange + every hook)"* to *"a daemon graphql-ws client for the streams that are
intrinsically streams"* — which is the irreducible, correct minimum.

## 4. Architecture gate

- **Owner map.** The **operator addon** owns the daemon transport (endpoint + minted token +
  its provider). `@refinedev/hasura` (via `createAngeeHasuraDataProvider`) owns the
  request/response dialect — reused unchanged because the daemon already speaks it. The
  composer (`packages/app/create-app.tsx`) owns the single `<Refine>` root and its provider
  map. Refine owns the data hooks the views call.
- **Sibling inventory.** The console/public schemas already register Hasura data providers via
  `createAngeeHasuraDataProviders` and consume them through `useList`/`useAuthoredQuery`/
  `useActionMutation` (`packages/ui/src/data/`). The operator becomes a third instance of the
  *same* pattern — not a new one. The authored-operation mutation path
  (`packages/ui/src/data/authored-hooks.tsx`) is the exact template for the command RPCs.
- **Dependency check.** No new library. We *remove* a use of urql's query/mutation surface;
  the daemon graphql-ws subscription transport stays (Phase 1) and can later drop to a thin
  `graphql-ws` hook (Phase 2), retiring the `urql` dependency from the operator web package.
- **Thin-caller check.** Operator views stop importing urql and stop hand-merging one-shot +
  push; they call Refine hooks with `dataProviderName:"operator"`. The provider owns transport;
  views declare intent.
- **Deletion check.** This is net-negative: deletes `operator-client.ts` (urql client +
  cacheExchange + fetchExchange wiring), the `useOperatorAction` mutation wrapper, the direct
  urql `useQuery` in two views, and (Phase 2) the urql dependency itself. The only addition is
  a small `bearerAuthFromGetter` (§5) and a token-store/gate refactor of the existing
  `OperatorTransportProvider`.
- **Naming.** One name: the new data provider is `operator` (matching its `dataProviderName`),
  consistent with `console`/`public`. No synonyms ("daemon provider" stays prose, not code).

## 5. The real blocker and its resolution

**Blocker:** the root `<Refine dataProvider>` is composed once in `create-app.tsx` from static
schema configs, but the daemon needs a **per-user minted bearer that rotates ~every 15 min**
(`transport.tsx:53`), and its `{ endpoint, token }` arrive from a runtime console
`OperatorConnection` query.

**Resolution — late-bound token via a getter:**

1. Add to `packages/refine/src/transport-auth.ts` (export from `@angee/refine`):
   ```ts
   /** Bearer auth whose token is read per-request, so client rebuilds aren't needed on rotation. */
   export function bearerAuthFromGetter(getToken: () => string | null): AuthFetch {
     return (baseFetch) => (input, init) => {
       const headers = new Headers(init?.headers);
       const token = getToken();
       if (token) headers.set("Authorization", `Bearer ${token}`);
       return baseFetch(input, { ...init, headers });
     };
   }
   ```
   The client is built once with `auth: bearerAuthFromGetter(getToken)`; rotation only updates
   the value the getter reads — no client rebuild, no re-registration.

2. Operator addon owns a small token store (module-scoped singleton, populated by React):
   ```ts
   // operator/web/src/data/operator-token.ts
   let token: string | null = null;
   export const operatorToken = { get: () => token, set: (t: string | null) => { token = t; } };
   ```

3. Build the `operator` data provider once with
   `createAngeeHasuraDataProvider({ url: operatorEndpoint, auth: bearerAuthFromGetter(operatorToken.get) })`
   and register it in the root `<Refine dataProvider>` map (§7, Open decision A picks the seam).

4. Refactor `OperatorTransportProvider` → **`OperatorTokenGate`**: keep the existing console
   `OperatorConnection` query + 15-min refresh + loading/not-configured/error gating
   (`transport.tsx:105-178`), but instead of building a urql client it writes the minted token
   into `operatorToken.set(...)` and (Phase 1) builds only the graphql-ws subscription client.
   Operator routes already wrap their section in this provider (`operator/web/src/index.ts:40`),
   so the token is populated before any operator hook runs; outside `/operator/*` nothing calls
   the operator provider.

**Endpoint note.** The endpoint is the same-origin default `/operator/graphql`
(`ANGEE_OPERATOR_GRAPHQL_ENDPOINT`). Treat it as the static provider URL. The rare
custom-endpoint deployment (connection query returns a different endpoint) is a documented
follow-up: either pass the endpoint as a host schema config, or expose a `setEndpoint` on the
provider client and call it from the gate on first resolve. Do **not** block Phase 1 on it.

## 6. Plan (phased)

**Phase 0 — primitive (tiny, no behavior change).**
- Add `bearerAuthFromGetter` to `transport-auth.ts`; export from `packages/refine/src/index.ts`;
  unit test it (mirrors `bearerAuth`).

**Phase 1 — register the provider; migrate request/response; kill direct urql in views.**
- Add `operator/web/src/data/operator-token.ts` (store) and the `operator` data provider
  factory.
- Wire `operator` into the root `<Refine dataProvider>` map (seam per §7-A).
- Refactor `OperatorTransportProvider` → `OperatorTokenGate` (token store + ws-only client;
  drop the urql query/mutation client).
- Migrate mutations: replace `useOperatorAction` call-sites with `useCustomMutation`
  (commands) / `useCreate`/`useDelete` (the `insert_*_one`/`delete_*_by_pk` docs), all
  `dataProviderName:"operator"`. Preserve the post-mutation snapshot refetch
  (`run-action.ts:26` `useRunDaemonAction`).
- Migrate the two one-shot queries (`ServiceDetail.tsx` endpoint, `logs.tsx` history) to
  `useAuthoredQuery`/`useCustom` on the `operator` provider. **Removes the direct
  `import { useQuery } from "urql"` from both views — the original finding.**
- Delete `operator-client.ts`'s fetch/cache exchanges (keep only the ws subscription transport,
  relocated into the gate or a `daemon-subscription` client).
- Verify: operator console loads, panes still live-update via the snapshot push, mutations fire
  + refetch, logs history + tail still work.

**Phase 2 — optional, retire urql entirely from the operator addon.**
- Replace the urql subscription host (`useDocumentSubscription`/`useOperatorSubscription`,
  `document-subscription.ts`) with a thin `graphql-ws`-backed hook (the daemon already uses
  `graphql-ws`; `@angee/refine` exports `graphQLWebSocketUrl`, `isFatalGraphQLWsClose`).
- Optionally move the snapshot first-paint `SNAPSHOT_QUERY` to `useAuthoredQuery` on the
  `operator` provider, keeping the live push on ws and the existing "whichever updated most
  recently" merge (`transport.tsx:255-267`).
- Drop `urql` from `operator/web/package.json`. This is the full DRY payoff.

## 8. Risks / out of scope / done

- **Risk — single root liveProvider.** Confirmed constraint; mitigated by keeping daemon
  subscriptions on their own ws transport (not a Refine liveProvider). No change to the console
  liveProvider.
- **Risk — token race on first paint.** The gate already blocks the subtree until
  `{ endpoint, token }` resolve; operator hooks run only inside the gated subtree.
- **Out of scope.** Daemon SDL changes (none needed); the snapshot-push model (kept);
  per-resource live queries (rejected, §3); custom-endpoint deployments (documented follow-up,
  §5).
- **Done when.** No `import … from "urql"` remains in operator **view** code; mutations + the two
  one-shot queries run through `dataProviderName:"operator"`; panes still live-update via the
  snapshot push; logs history+tail+socket work; `docs/stack.md` operator/daemon rows updated to
  say the daemon is consumed as a Refine `operator` provider with subscriptions on its own ws
  transport (replacing the "quarantine until non-app owner" wording, which this resolves).

## 9. Open decisions for the architect

- **A. Provider-contribution seam (pick one).**
  - *Addon-contributed provider (owner-correct):* add a `dataProviders` contribution to the
    `defineBaseAddon`/`composeAddons` seam so the operator addon contributes its `operator`
    provider (with its token store + gate), merged into the root map by `create-app`. Generalizes
    to any addon with its own backend. More work; the principled target.
  - *Host schema config (lighter MVP):* the host declares `schemas: { operator: { url:
    "/operator/graphql", auth: bearerAuthFromGetter(operatorToken.get) } }`. Faster, but couples
    the host config to the operator addon's token store.
  - **Recommendation:** addon-contributed seam if the operator surface will grow or other addons
    will gain backends; host schema as an interim if not.
- **B. Snapshot first-paint query** — move to Refine (Phase 2) or leave on ws? Low stakes; leave
  for last.
- **C. Is the consistency worth it now?** The win is real (uniform hook surface, urql deleted),
  but Phase 1 touches ~12 files for a surface that is stable. If the operator console is frozen,
  this can sit as a documented, ready plan rather than execute immediately.
