# Code Review — Angee framework core (`src/angee/base/`)

## Summary

The framework core is broadly well-structured: discovery, planning, emission,
and serving are cleanly separated, and most behavior is correctly placed on
`AppConfig`, models, managers, and querysets as the backend guidelines demand.
The single biggest structural problem is the **base addon's own migration**:
`Resource` is an abstract source model, yet a concrete, hand-committed
`src/angee/base/migrations/0001_initial.py` lives in the installed-package
source tree with a non-normalized wall-clock header — the base app is the one
addon whose migrations escape the composer's runtime/normalize pipeline,
breaking the "change the source, not the artifact" and byte-determinism rules
that the rest of the build enforces. Beyond that, the code repeatedly re-decodes
the same shapes from outside their owners (label parsing, `from_public_id`
resolution, model-reference formatting, Strawberry surface introspection),
violating the "Find the owner" constitution in several small but consistent
ways. Idiomatic Django ownership is mostly respected, but a cluster of loose
helpers that branch on or re-read object internals want to be methods on the
owning class. The code mostly lives up to its constitution; the violations are
concentrated and fixable.

## Findings

### 1. Base addon migration is a checked-in artifact outside the composer pipeline

- **Dimension(s)**: Architecture, Consistency, DRY
- **Location**: `src/angee/base/migrations/0001_initial.py:1`; pipeline at
  `src/angee/base/compose/pipeline.py:72-79`; exclusion at
  `src/angee/base/settings.py:100-128`
- **Severity**: Critical
- **Problem**: `Resource` is `abstract = True`
  (`src/angee/base/resources/models.py:52`); the concrete model is emitted into
  `runtime/base/models.py` with `app_label = "base"` and loaded by
  `BaseConfig.import_models()` (`src/angee/base/apps.py:346-355`). Because
  `_addon_config_classes` excludes `BaseConfig`
  (`src/angee/base/settings.py:111-114`), `_migration_modules` never adds a
  `"base"` entry, so `MIGRATION_MODULES` does **not** redirect `base` to
  `runtime.base.migrations`. But `base` *is* in `plan.labels` (Resource is an
  abstract `AngeeModel`, so `get_model_classes()` returns it), so
  `call_command("makemigrations", *plan.labels, ...)`
  (`pipeline.py:73-78`) writes the `base` migration into the *installed package
  source tree* `src/angee/base/migrations/`, not into `runtime/`. The header
  carries a wall-clock timestamp (`# Generated by Django 6.0.5 on 2026-05-30
  18:40`) that `_normalize_migration_headers` never strips, because that step
  only scans `runtime/<label>/migrations` (`emission.py:580-591`). The result:
  one addon's migrations live in source, are non-deterministic, and are not
  drift-checked the way every other addon's are. This violates the
  Django-Native Rule (`runtime/`, migrations are output — change the source),
  "Build output must be byte-deterministic", and "never use wall-clock time …
  in emitted artifacts".
- **Recommendation**: Treat `base` like every other emitted addon. Either
  include `BaseConfig` in `_migration_modules` so `base` migrations are
  redirected to `runtime.base.migrations` and normalized, or stop excluding it
  from the migration-module map while still installing it explicitly. Then
  delete the committed `src/angee/base/migrations/0001_initial.py`. The
  exclusion comment at `settings.py:106-107` only justifies skipping the
  *INSTALLED_APPS* duplication, not the migration-module redirect — split those
  two concerns.

### 2. `app_label.Model` label parsing is reimplemented in three places

- **Dimension(s)**: DRY, Code decomposition, Find-the-owner
- **Location**: `src/angee/base/mixins/models.py:55-66` (`normalize_model_label`);
  `src/angee/base/resources/managers.py:210-227` (`_model_for_label`);
  `src/angee/base/resources/widgets.py:145-149` (`_model_from_label`)
- **Severity**: High
- **Problem**: All three call `make_model_tuple` and then either format or
  resolve the result, each with its own error handling and message
  (`ImproperlyConfigured` vs raw). This is the same fact — "parse/resolve a
  Django model label" — decoded in three spots, contradicting "One source of
  truth per fact" and the DRY section's "Same shape in three places: extract
  the smallest boring primitive". `_model_from_label` (widgets) has *no*
  error handling at all, so a bad stored `target_model` raises a bare
  `LookupError` instead of a load error.
- **Recommendation**: Extract one module-level resolver (e.g. in a small
  `resources` or shared module) `resolve_model(label) -> type[models.Model]`
  that normalizes and resolves with one consistent error, and have all three
  callers use it. Keep `normalize_model_label` on `AngeeModel` only if the
  normalized *string* (not the class) is genuinely needed there; otherwise fold
  it in too.

### 3. `from_public_id` resolution is decoded from outside the owner three times

- **Dimension(s)**: Find-the-owner, DRY, Code decomposition
- **Location**: `src/angee/base/resources/widgets.py:152-169`
  (`_instance_from_public_id`, `public_id`); `src/angee/base/graphql/crud.py:127-142`
  (`_resolve_for_delete`); owner at `src/angee/base/mixins/models.py:124-161`
- **Problem**: Three call sites do `getattr(model, "from_public_id", None)` /
  `isinstance(instance, AngeeModel)` and fall back to
  `_default_manager.filter(pk=...)`. This is the exact smell the constitution
  names: "a function that takes an object and inspects it to decide something".
  The owner (`AngeeModel`/`SqidMixin`) already defines `from_public_id` and
  `public_id`; the framework should rely on the method rather than re-probe for
  its existence from outside. The fallback only exists because non-`AngeeModel`
  models lack the method — but every composed source model *is* an `AngeeModel`,
  so the fallback is dead defensiveness around the framework's own contract.
- **Severity**: High
- **Recommendation**: Make `from_public_id`/`public_id` a guaranteed part of the
  composed-model contract (they already live on `AngeeModel`), then call them
  directly. If a true non-Angee target is possible (e.g. `auth.User`), put the
  resolution behind one shared classmethod or a single helper, not three
  copies, and type it on the contract rather than `getattr`-probing.

### 4. Dotted model-reference formatting is duplicated and re-decoded

- **Dimension(s)**: DRY, Find-the-owner
- **Location**: `src/angee/base/mixins/models.py:97-101` (`get_model_reference`);
  `src/angee/base/compose/emission.py:409-420` (`_model_reference`);
  `src/angee/base/graphql/subscriptions.py:276-279` (`_model_label`)
- **Severity**: Medium
- **Problem**: `get_model_reference` returns `f"{module}.{__name__}"`,
  `_model_reference` does the same with a `getattr` fallback for non-Angee
  bases, and `_model_label` builds `f"{app_label}.{object_name}"`. Three
  near-identical "name a model" formatters, two of which (`_model_reference`,
  `_model_label`) re-decode `_meta` from outside. `emission._model_reference`
  and `_declared_composition_fields` (`emission.py:381-420`) both justify their
  `getattr`-dispatch as "intentional polymorphism over two shapes" — but
  polymorphism is what classes are for; the comment is describing the smell, not
  excusing it.
- **Recommendation**: Standardize on the owner: `Model._meta.label` /
  `label_lower` is Django's native dotted reference. Where a plain Django mixin
  base lacks the Angee method, give the extension base a tiny shared mixin or
  use `_meta.label` directly instead of dual-shape `getattr` dispatch.

### 5. `_surface_name` and Strawberry-surface introspection duplicated across graphql modules

- **Dimension(s)**: DRY, Code decomposition
- **Location**: `src/angee/base/graphql/crud.py:187-190` and
  `src/angee/base/graphql/schema.py:135-138` (identical `_surface_name`);
  `crud.py:170-178` (`_model` via `__strawberry_django_definition__`) vs
  `schema.py:124-132` (`_surface_field_names` via `__strawberry_definition__`)
- **Severity**: Medium
- **Problem**: `_surface_name` is byte-identical in two modules. Both modules
  reach into Strawberry private dunder attributes (`__strawberry_definition__`,
  `__strawberry_django_definition__`) to decode a surface from outside. Per
  stack.md, strawberry-django owns "GraphQL types"; the framework should ask the
  library, not re-decode its internals in scattered helpers.
- **Recommendation**: Move the shared introspection into one place in
  `graphql/` (a single `surface.py`-style helper module or functions in
  `schema.py` imported by `crud.py`). Collapse the two `_surface_name`
  definitions into one.

### 6. `crud.py` is not formatted to the repo standard and uses argv-style call layout

- **Dimension(s)**: Consistency, Readability
- **Location**: `src/angee/base/graphql/crud.py:68-77`
- **Severity**: Medium
- **Problem**: `uv run ruff format --check src/angee/base/graphql/crud.py`
  reports the file *would be reformatted*. The `add("create", node,
  strawberry_django.mutations.create(create, permission_classes=...))` calls
  trail their closing args onto the next line in a hand-wrapped style ruff
  rejects, unlike every other file in `base`. The backend guidelines require
  `ruff` to pass; this is the only non-conforming source file under review.
- **Recommendation**: Run `uv run ruff format` on the file; the inner
  `strawberry_django.mutations.*` calls should be assigned to locals first for
  readability.

### 7. `_running_angee_build` inspects `sys.argv` to branch composition behavior

- **Dimension(s)**: Find-the-owner, Architecture, Readability
- **Location**: `src/angee/base/apps.py:36-46`, used at `apps.py:350`
- **Severity**: Medium
- **Problem**: `import_models()` decides whether to import the previous runtime
  by checking `sys.argv[1:3] == ["angee", "build"]`. This is a fragile global
  signal: it breaks under `manage.py angee build`, programmatic `call_command`,
  test harnesses, or any wrapper that doesn't put `angee build` at exactly
  argv[1:3] — and the build pipeline (`pipeline.run`) is itself callable with an
  explicit `addons=` argument, bypassing argv entirely. The docstring concedes
  "argv is the only signal available", which is the "I don't fully control this"
  red flag. The owner of "is a build in progress" is the build pipeline, not
  argv.
- **Recommendation**: Have the build set an explicit, process-local flag (e.g. a
  module-level sentinel in `compose.pipeline` toggled around `run()`, or a
  setting/env the build controls) and have `import_models()` read that, so the
  signal is owned by the code that creates the condition rather than sniffed
  from the command line.

### 8. Module-level mutable global `_active_ledger_model` with `set_ledger_model`

- **Dimension(s)**: Architecture, Find-the-owner
- **Location**: `src/angee/base/resources/widgets.py:16-31`; toggled in
  `src/angee/base/resources/managers.py:46-56, 91-117`
- **Severity**: Medium
- **Problem**: A global `_active_ledger_model` is set/cleared with
  `global ... # noqa: PLW0603` around load/validate to tell widgets which
  concrete `Resource` model to use. This is hidden, non-reentrant cross-module
  state (spaghetti-code red flag: "state mutated from far-away places"); two
  concurrent loads or a forgotten `finally` corrupts it, and widgets fall back
  to `apps.get_model("base", "Resource")` otherwise. The widget's need is "which
  ledger model" — a fact the resource/loader already knows and could pass.
- **Recommendation**: Thread the concrete ledger model (or its manager) through
  the resource/widget construction explicitly — the `AngeeResource` already
  carries `entry`; pass the ledger model into the resource or widget factory so
  widgets ask their resource instead of a module global.

### 9. `globals()` is not passed in, but the host pattern lives in code untested by base

- **Dimension(s)**: Consistency, Readability
- **Location**: `src/angee/base/settings.py:16-44`
- **Severity**: Low
- **Problem**: `compose_defaults` correctly stays a pure-ish function returning a
  mapping (good, matches the settings rule), but it performs a documented
  `sys.path.insert` side effect inside what is otherwise framed as a pure
  helper. The docstring calls it a "Required side effect", which is honest, but
  it means the function is not the pure function the guideline describes
  ("Settings helpers are pure functions of their arguments … do not read the
  environment"). It does not read the environment, but it mutates global
  interpreter state.
- **Recommendation**: Either move the `sys.path` insertion to the host (it owns
  where runtime lives, per the guideline) and keep `compose_defaults` purely a
  mapping builder, or rename to signal the effect. Returning the path to add and
  letting the host insert it in "one visible step" matches the stated contract.

### 10. `from_public_id` / `public_id` typed as `Any` weakens the contract

- **Dimension(s)**: Readability, Naming/typing
- **Location**: `src/angee/base/mixins/models.py:131,158`
- **Severity**: Low
- **Problem**: `from_public_id(cls, value: str) -> Any | None` returns `Any`,
  defeating mypy on every caller and pushing callers toward the `getattr`-probe
  pattern in finding #3. The natural return is `Self | None`.
- **Recommendation**: Type it `-> Self | None` (PEP 673) so callers get the
  concrete model type and the framework contract is self-documenting per
  "Framework contracts should be self-explaining in code".

### 11. `_capitalize` reimplements `str.title`/`capitalize` semantics by hand

- **Dimension(s)**: Readability, DRY (reinvent-the-wheel)
- **Location**: `src/angee/base/graphql/crud.py:181-184`
- **Severity**: Low
- **Problem**: `singular[:1].upper() + singular[1:]` is a hand-rolled
  capitalize. Django ships `django.utils.text` helpers and the model already
  exposes proper names. Minor, but the guidelines explicitly flag "rewriting a
  utility that … already provides".
- **Recommendation**: Use the model's own `verbose_name`/`object_name` where a
  display label is wanted, or `str.capitalize()` semantics via a stdlib path;
  derive field names from `_meta` rather than string-casing `model_name`.

## Patterns & inconsistencies

- **Decode-from-outside is the dominant recurring smell.** Findings #2, #3, #4,
  #5, #8 are the same constitution violation in five guises: helpers that
  `getattr`-probe or re-read an object's `_meta`/dunder shape to decide
  something the owner already knows. The codebase even *names* this in comments
  ("intentional polymorphism over those two shapes", `emission.py:389`,
  `emission.py:412`) — those comments are flags that the behavior belongs on a
  class, not a loose function. A refactor pass that pushes label resolution,
  model-reference formatting, and public-id resolution onto the owning model /
  manager would remove the bulk of the duplication at once.

- **The base addon is special-cased in ways that leak.** Finding #1 (migration
  in source tree) and the `BaseConfig` exclusion in `settings.py` both stem from
  base being installed explicitly while every other addon goes through
  discovery. The seam treats base differently, and that difference escapes the
  determinism/normalize guarantees the framework otherwise enforces. Base should
  flow through the same emission/normalize path as any addon.

- **Build vs runtime signaling is implicit.** `_running_angee_build` (argv) and
  `_active_ledger_model` (mutable global) are both implicit channels for "what
  phase/context am I in". The framework's own value proposition is explicit
  build-time composition; these two spots reach for runtime sniffing instead.

- **Tooling is not green on the reviewed tree.** `ruff check` reports 5 errors
  (all in the stray migration) and `ruff format --check` would reformat
  `crud.py`. The backend "Checks" gate is currently failing on `base`.

- **Error-handling consistency is good but not uniform.** Most label/format
  failures raise `ImproperlyConfigured` with an owner-qualified message;
  `widgets._model_from_label` is the outlier that lets a bare `LookupError`
  escape (finding #2).

## Top 5 recommendations

1. Route the `base` addon's migrations through the same `runtime/` emission and
   header-normalization path as every other addon, then delete the committed
   `src/angee/base/migrations/0001_initial.py` (fixes determinism + the failing
   ruff check).
2. Extract one `resolve_model(label)` primitive and replace the three
   label-parsing reimplementations (`normalize_model_label`, `_model_for_label`,
   `_model_from_label`).
3. Make `from_public_id`/`public_id` a typed (`Self | None`) part of the
   composed-model contract and call it directly, removing the three
   `getattr`-probe-and-fallback copies.
4. Replace the `sys.argv` build check and the `_active_ledger_model` global with
   explicit phase/context owned by the build pipeline and passed to the widgets.
5. Run `ruff format` on `crud.py`, dedupe `_surface_name`/surface-introspection
   into one graphql helper, and consolidate dotted model-reference formatting on
   `_meta.label`.
