### Summary

The Angee backend codebase demonstrates a sophisticated and well-architected approach to building a composable Django application. The build-time code generation is robust, with strong safety checks against misconfiguration and data loss. The data resource loading subsystem is powerful, featuring idempotency and dependency management. However, the most critical issue found is a high-severity Server-Side Request Forgery (SSRF) vulnerability in the remote resource fetching mechanism, which uses `urllib.request.urlopen` insecurely. Additionally, there are medium-severity performance concerns related to potential N+1-type problems in authorization checks and potential event-loop blocking in the GraphQL subscription handling.

### Findings

1.  **SSRF Vulnerability in Remote Resource Fetching**
    *   **Lens(es)**: Security
    *   **Location**: `src/angee/resources/fetch.py:23`
    *   **Severity**: Critical
    *   **Problem**: The `fetch_url` function uses `urllib.request.urlopen` to download remote resources. While it checks for `http` or `https` schemes initially, it does not safely handle HTTP redirects. A malicious server could redirect the request to a different protocol (e.g., `file://`), allowing an attacker to read arbitrary files from the server's filesystem. This is a classic SSRF vulnerability. The code itself contains a `noqa: S310` comment, indicating a linter has already flagged this potential issue.
    *   **Recommendation**: Replace the use of `urllib.request.urlopen` with a library that handles redirects securely by default, such as `requests`. If adding a new dependency is not an option, the redirect handling must be done manually, ensuring that the scheme of the redirected URL is validated against the allowed list (`http`, `httpshttps`) before the next request is made. Given the risk, disallowing HTTP and only permitting HTTPS would be a reasonable interim measure.

2.  **Potential Event-Loop Blocking in GraphQL Subscriptions**
    *   **Lens(es)**: Async / ASGI
    *   **Location**: `src/angee/base/graphql/subscriptions.py:48`
    *   **Severity**: Medium
    *   **Problem**: The `async def resolve` function in the `changes` subscription calls `_gate_event`, which is a synchronous function that performs authorization checks via `ChangeReadGate`. This synchronous, potentially CPU-intensive work is executed directly within the async function without being offloaded to a thread pool. This can block the event loop, degrading the performance and responsiveness of the entire ASGI application, especially under load.
    *   **Recommendation**: Wrap the synchronous call to `_gate_event` in `asgiref.sync.sync_to_async`. The call should be changed from `event = _gate_event(...)` to `event = await sync_to_async(_gate_event, thread_sensitive=True)(...)`.

3.  **Potential N+1 Problem in Authorization Read Gate**
    *   **Lens(es)**: Query Performance, Authorization
    *   **Location**: `src/angee/base/access.py:64`
    *   **Severity**: Medium
    *   **Problem**: The `ChangeReadGate._redact` method iterates over `changed_fields` and calls `self._can_read_field` for each field. This in turn calls `check_field_access` for each individual field. If the `rebac` backend (`django-zed-rebac`) is configured to use a remote service like SpiceDB, this will result in a separate network request for every field in the update payload, creating a significant performance bottleneck.
    *   **Recommendation**: Investigate if the `rebac` library provides a bulk-check API to verify permissions for multiple fields in a single call. If such an API exists, refactor the `_redact` method to gather all fields that need checking and use the bulk API to reduce the number of calls to the authorization backend to one.

4.  **Missing Transaction in Delete Mutation**
    *   **Lens(es)**: Transactions & integrity
    *   **Location**: `src/angee/base/graphql/crud.py:141`
    *   **Severity**: Low
    *   **Problem**: The `delete` resolver generated by `_delete_resolver` first calls `DeletionPreview.from_instance()` to gather which objects will be deleted, and then calls `instance.delete()`. These two operations are not wrapped in a transaction. In a highly concurrent system, it is possible for the state of the database to change between the preview being generated and the delete being executed, leading to a different outcome than what was previewed.
    *   **Recommendation**: Wrap the body of the `delete` resolver in `django.db.transaction.atomic`. This will ensure that the preview and the actual deletion occur within a single, consistent database transaction.

### Patterns & inconsistencies

*   **Excellent Transaction Safety**: The codebase consistently demonstrates strong practices for data integrity. The resource loading subsystem is wrapped entirely in `transaction.atomic` (`src/angee/resources/managers.py:84`), and signal handlers correctly use `transaction.on_commit` to defer actions until after a transaction succeeds (`src/angee/base/signals.py:93`). This is a commendable pattern.
*   **Robust Build-Time Safety**: The `angee.compose` package is very well-written, with critical safety checks to prevent incorrect or destructive operations. The `_ensure_cleanable` check in `src/angee/compose/runtime.py` is a prime example of preventing accidental data loss by verifying a sentinel file before cleaning a directory.
*   **Asynchronous Code Correctness**: While one instance of potential event-loop blocking was found, other parts of the codebase show correct async/sync interoperability. For example, `src/angee/base/signals.py` correctly uses `async_to_sync` to call `group_send` from a synchronous signal handler. This suggests the issue is an isolated oversight rather than a systemic problem.

### Top recommendations

1.  **Fix the SSRF vulnerability in `src/angee/resources/fetch.py` immediately** by replacing `urllib.request.urlopen` with a secure alternative like the `requests` library.
2.  **Prevent event-loop blocking in GraphQL subscriptions** by wrapping the synchronous `_gate_event` call in `sync_to_async` in `src/angee/base/graphql/subscriptions.py`.
3.  **Investigate and implement a bulk authorization check** in `src/angee/base/access.py` to mitigate the potential N+1 performance issue with field-level security checks.
4.  **Wrap the delete mutation logic** in `src/angee/base/graphql/crud.py` within `transaction.atomic` to ensure atomicity between the deletion preview and the delete operation.
