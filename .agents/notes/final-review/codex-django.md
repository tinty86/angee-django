### Summary
No critical findings. The riskiest issue is the cascade delete preview: it misreads Django 6 `Collector` internals and can crash or underreport affected rows before the CRUD delete mutation runs. The resource loader is generally stack-aligned, but several paths validate too late, query per row, or allow ambiguous xref resolution. Verification: `mypy src/`, `pytest`, and `examples/notes-angee/manage.py angee build --check` passed; `ruff check .` failed import ordering, including `src/angee/resources/management/commands/resources.py:8`.

### Findings
1. **Deletion preview misreads Django collector state**; **Lens(es)**: Django runtime correctness, transactions/integrity; **Location**: `src/angee/base/deletion.py:52`, `src/angee/base/deletion.py:53`, `src/angee/base/deletion.py:56`, `src/angee/base/deletion.py:59`, `src/angee/base/graphql/crud.py:134`; **Severity**: High; **Problem**: `Collector.field_updates` is keyed by `(field, value)` with list values in Django 6, but the code treats it as `model -> dict`, so `SET_NULL`/`SET_DEFAULT` cascades crash with `AttributeError`. It also catches `ProtectedError` but not `RestrictedError`, and ignores `collector.fast_deletes`, so delete previews can fail or undercount rows. **Recommendation**: Count `field_updates` by `field.model`, catch both `ProtectedError` and `RestrictedError`, include `fast_deletes` counts, and add regression tests for `SET_NULL`, `RESTRICT`, and fast-delete cascades.

2. **`angee clean` can leave runtime output permanently uncleanable**; **Lens(es)**: migrations, generated runtime safety; **Location**: `src/angee/compose/runtime.py:112`, `src/angee/compose/runtime.py:118`, `src/angee/compose/runtime.py:349`; **Severity**: High; **Problem**: `clean()` preserves `*/migrations/` but deletes the root generated sentinel in `__init__.py`; the next `emit()`/`reset()` calls `_ensure_cleanable()`, sees children but no sentinel, and refuses to rebuild. **Recommendation**: Leave or rewrite the sentinel during clean, or teach `_ensure_cleanable()` to accept a migrations-only generated layout after verifying every remaining path is preserved migration output.

3. **WebSocket GraphQL does not enter the REBAC request context**; **Lens(es)**: async/ASGI, authorization; **Location**: `src/angee/base/consumers.py:24`, `src/angee/base/consumers.py:25`, `src/angee/base/graphql/subscriptions.py:32`, `src/angee/base/access.py:43`; **Severity**: Medium; **Problem**: the consumer stores the actor only in Strawberry context, while django-zed-rebac querysets and write signals use the ambient actor context installed by `ActorMiddleware`. Non-subscription operations over GraphQL WS can therefore fail strict-mode ORM access, and subscription read checks call the synchronous REBAC backend directly from the async generator. **Recommendation**: Add Channels/GraphQL execution middleware that brackets WS operations in the REBAC actor/evaluator/zookie context, and run synchronous backend checks through `sync_to_async(..., thread_sensitive=True)` unless the library provides an async check API.

4. **Resource xref resolution is unindexed and first-row-wins**; **Lens(es)**: query performance, data integrity; **Location**: `src/angee/resources/widgets.py:102`, `src/angee/resources/widgets.py:109`, `src/angee/resources/models.py:50`, `src/angee/resources/managers.py:191`; **Severity**: Medium; **Problem**: `resolve_xref()` filters by `xref` plus exact/suffix `source_addon`, orders, and returns `.first()`, but the only DB invariant is unique on `(source_addon, source_path, xref, target_model)`. Collisions across source paths, tiers, or old ledger rows are possible, and the query has no supporting xref-first index. **Recommendation**: Make the xref contract match the DB invariant: either enforce unique `(source_addon, xref)` and index it, or include source/tier/model in the reference syntax and resolve deterministically without suffix scans.

5. **Resource loading performs per-row ledger and target queries**; **Lens(es)**: ORM/query performance, stack-library usage; **Location**: `src/angee/resources/managers.py:85`, `src/angee/resources/loader.py:114`, `src/angee/resources/loader.py:197`, `src/angee/resources/loader.py:220`, `src/angee/resources/widgets.py:61`; **Severity**: Medium; **Problem**: each imported row can query the ledger, resolve target objects, and `update_or_create()` a ledger row; FK/M2M widgets add more per-value lookups. Large resource files will degrade into N+1 behavior. **Recommendation**: Preload ledger rows per entry/model, cache target objects by `(model, public_id)`, cache xref resolutions for the import run, and batch ledger writes where import-export row semantics allow it.

6. **`resources validate` does not validate import-export row cleaning**; **Lens(es)**: stack-library usage, data integrity; **Location**: `src/angee/resources/managers.py:49`, `src/angee/resources/managers.py:54`, `src/angee/resources/managers.py:92`; **Severity**: Medium; **Problem**: validation only checks headers with `before_import()`, so bad FK xrefs, model validation failures, required fields, and widget cleaning errors can pass `resources validate` and fail later in `resources load`. **Recommendation**: Run `resource.import_data(..., dry_run=True, raise_errors=True, rollback_on_validation_errors=True)` inside a rollback-only transaction under `system_context`, or rename the command to make clear it is header/file validation only.

7. **Structured resource `fields` can override reserved row identity**; **Lens(es)**: data integrity, mass-assignment; **Location**: `src/angee/resources/entries.py:373`, `src/angee/resources/entries.py:379`; **Severity**: Medium; **Problem**: when a structured row uses `fields`, reserved keys are copied through unchanged, and `dataset_row` merges `self.values` after the canonical `_xref`. A nested `fields._xref` can replace the row identity used by import-export. **Recommendation**: Reject `RESERVED_ROW_KEYS` inside `fields`, or merge reserved loader keys after model values so metadata cannot be overridden.

8. **String `depends_on` is split into characters**; **Lens(es)**: resource runtime correctness; **Location**: `src/angee/base/apps.py:336`; **Severity**: Low; **Problem**: manifest normalization calls `tuple(entry["depends_on"])`, so `"seed.csv"` becomes `("s", "e", "e", "d", ...)` instead of one dependency. **Recommendation**: Treat a string as a single dependency or reject strings with a clear `ImproperlyConfigured` error.

### Patterns & inconsistencies
Resource loading repeatedly falls back to row-by-row ORM work instead of building import-run caches around django-import-export. Generated runtime safety is strict, but clean/build/schema lifecycle paths are not consistently excluded or preserved. The ASGI path handles a Strawberry context actor but does not consistently install the django-zed-rebac ambient context used by managers/signals. `src/angee/resources/entries.py:11` imports PyYAML even though `docs/stack.md` assigns tabular import/export ownership to django-import-export + tablib and has no PyYAML owner row.

### Top recommendations
1. Fix `DeletionPreview.from_instance()` against Django 6 collector shapes before relying on CRUD delete in real schemas.
2. Make runtime clean idempotent with preserved migrations, then add a clean-followed-by-build regression test.
3. Define and enforce one xref uniqueness contract in both the resource ledger schema and resolver.
4. Move resource validation and load performance closer to django-import-export’s dry-run and batch-friendly primitives.
5. Add an ASGI REBAC context wrapper for GraphQL WS execution.
